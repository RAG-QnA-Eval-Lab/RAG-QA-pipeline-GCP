# 멀티클라우드 전략: GCP ↔ AWS 데이터 공유

## 배경

| 프로젝트 | 인프라 | 범위 |
|----------|--------|------|
| **개인 프로젝트** (졸업논문 + PyCon) | GCP | 전체 RAG 파이프라인 (수집 → 검색 → 생성 → 평가) |
| **캡스톤 팀 프로젝트** | AWS | 팀원들이 AWS 인프라로 서빙/추론 구축 |

데이터 수집 파이프라인과 QA 쌍 준비는 **두 프로젝트에서 공통**으로 필요한 작업이다.
GCP에서 수집/적재를 완료하고, AWS 쪽에서 해당 데이터를 가져다 쓰는 구조로 중복 작업을 제거한다.

---

## 아키텍처

```
┌─────────────────────────────────┐     ┌──────────────────────────────────┐
│          GCP (개인)              │     │          AWS (캡스톤 팀)           │
│                                 │     │                                  │
│  collectors ──→ GCS (원본 JSON) │──→──│  S3 (동기화 or Signed URL)        │
│                   │             │     │    │                              │
│              chunker/embedder   │     │    ↓                              │
│                   │             │     │  팀 RAG 서빙 시스템               │
│              FAISS 인덱스 (GCS) │──→──│  S3에서 인덱스 로드               │
│                                 │     │                                  │
│  MongoDB (GCP VM)               │     │  팀 자체 메타데이터 관리          │
│    └─ (필요 시 Atlas 전환)      │──→──│  Atlas 공유 or 별도 DB 구축       │
│                                 │     │                                  │
│  QA 쌍 (data/eval/)            │──→──│  평가용 데이터셋 공유             │
└─────────────────────────────────┘     └──────────────────────────────────┘
```

## GCP → AWS 데이터 공유 방법

| 방법 | 설명 | 난이도 | 적합한 경우 |
|------|------|--------|------------|
| **GCS Signed URL** | GCS 객체에 대한 시간 제한 다운로드 URL 생성, AWS 쪽에서 curl/wget으로 다운로드 | 가장 쉬움 | 비정기적 공유, 소량 파일 |
| **GCS → S3 동기화** | Cloud Function 또는 Scheduler로 주기적으로 GCS → S3 복사 (gsutil/gcloud) | 쉬움 | 정기적 동기화, 자동화 필요 시 |
| **GCS S3 호환 API** | GCS가 S3 호환 엔드포인트 제공 — AWS SDK(boto3)로 GCS 직접 접근 가능 | 쉬움 | AWS 코드 변경 최소화 |
| **Storage Transfer Service** | GCP 관리형 서비스로 GCS → S3 자동 전송 | 중간 | 대용량, 스케줄 기반 전송 |

## 공유 대상 데이터

| 데이터 | 위치 (GCP) | 포맷 | 공유 방법 |
|--------|-----------|------|----------|
| 수집된 정책 원본 | `gs://rag-qna-eval-data/policies/` | JSON, PDF | S3 동기화 or Signed URL |
| FAISS 인덱스 | `gs://rag-qna-eval-data/index/` | `faiss.index` + `metadata.pkl` | S3에 복사 → 팀 서버에서 로드 |
| QA 평가 데이터셋 | `data/eval/qa_pairs.json` | JSON | Git 레포 공유 or S3 업로드 |
| 청크 데이터 | `gs://rag-qna-eval-data/chunks/` | JSON | 팀이 자체 임베딩 시 필요 |

## MongoDB 전략

- **개인 프로젝트**: GCP Compute Engine (e2-small)에 MongoDB 설치하여 사용
- **팀 공유 필요 시**: MongoDB Atlas 프리티어 (512MB)로 전환 — GCP/AWS 양쪽에서 동일 DB 접근 가능
- **캡스톤 팀 자체 메타데이터**: 팀에서 별도로 구성 (이 프로젝트 범위 밖)

---

## 현재 설계에 미치는 영향

**변경 불필요.** GCP 계획대로 구축한다. 캡스톤에서 데이터 공유가 필요한 시점에 동기화 레이어만 추가하면 된다.

수집 파이프라인의 핵심 로직(collectors)은 순수 Python 크롤링/API 호출 코드이므로 클라우드 종속성이 없다. 저장소 연결(GCS 업로드) 부분만 환경에 맞게 교체하면 파이프라인 자체를 AWS에서 실행하는 것도 가능하다.
